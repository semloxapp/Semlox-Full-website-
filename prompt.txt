# STRICT FIX — PRODUCTION LOGIN SESSION_COOKIE_ERROR

We deployed SemloX to Vercel.

Current issue:
- Public companies load correctly, so Supabase connectivity is working.
- Same login credentials work locally.
- In production, POST /api/auth/login returns 500.
- Response body is:

{
  "ok": false,
  "code": "SESSION_COOKIE_ERROR",
  "message": "Could not start your session. Please try again."
}

This means login likely reaches Supabase, but fails while creating or setting the `semlox_session` cookie.

CRITICAL RULES:
- Do NOT rewrite the auth system.
- Do NOT change database schema.
- Do NOT change Supabase tables.
- Do NOT change frontend UI.
- Do NOT remove cookie-based auth.
- Only fix `/api/auth/login` cookie/session creation safely.
- Keep local login working.

Current auth design:
- Primary session cookie name: `semlox_session`
- Cookie must be httpOnly
- Cookie must be secure in production
- Cookie must use sameSite lax
- Cookie must use path "/"
- Middleware validates this cookie
- Frontend relies on this cookie after login

Files to inspect:
- `app/api/auth/login/route.ts`
- `lib/auth.ts`
- `middleware.ts`

Main task:
Find exactly why `/api/auth/login` throws `SESSION_COOKIE_ERROR` in production.

Check these possible causes:

1. Cookie value too large
If the route stores the full Supabase session/user object in `semlox_session`, fix it.
Do NOT store full session object.
Do NOT store full user object.
Only store minimal data needed by middleware/helper:

{
  access_token,
  refresh_token,
  expires_at
}

Optionally include:
{
  user_id,
  email
}

But avoid full metadata.

Cookie should stay under browser cookie size limits.

2. Wrong cookie setting method
If the route uses `Response.json()` and then tries to set cookies incorrectly, fix it to use `NextResponse`.

Correct pattern:

```ts
import { NextResponse } from "next/server";

const response = NextResponse.json({
  ok: true,
  user: {
    id: user.id,
    email: user.email
  }
});

response.cookies.set("semlox_session", encodedSession, {
  httpOnly: true,
  secure: process.env.NODE_ENV === "production",
  sameSite: "lax",
  path: "/",
  maxAge: maxAgeSeconds
});

return response;




Do not use a pattern that works locally but fails on Vercel.

3. Encoding issue
   If encoding uses `btoa()` or browser-only APIs, replace with Node-safe encoding:

```ts
const encodedSession = Buffer.from(JSON.stringify(sessionPayload), "utf8").toString("base64url");
```

And decoding should match:

```ts
JSON.parse(Buffer.from(cookieValue, "base64url").toString("utf8"));
```

4. Runtime issue
   If the route is using Edge runtime and Node APIs like Buffer are not supported, either:

* remove `runtime = "edge"` from `/api/auth/login`, or
* use Web-compatible encoding safely.

Preferred:
Use Node.js runtime for auth login route if cookie/session encoding uses Buffer.

5. Add safe logs temporarily
   Add safe server logs around cookie creation.

Allowed logs:

* `[auth-login] request received`
* `[auth-login] supabase sign-in success true/false`
* `[auth-login] session exists true/false`
* `[auth-login] cookie payload keys`
* `[auth-login] encoded cookie length`
* `[auth-login] setting cookie`
* `[auth-login] cookie set success`
* `[auth-login] cookie error message`

Never log:

* password
* access token
* refresh token
* full session
* cookie value
* service role key
* authorization header

6. Improve error detail internally
   Keep user-facing message safe, but log exact server error in Vercel logs.

Return structured error:

```json
{
  "ok": false,
  "code": "SESSION_COOKIE_ERROR",
  "message": "Could not start your session. Please try again."
}
```

But server logs should show the real cause.

7. Success criteria
   After fix:

* POST /api/auth/login returns 200
* response body has `{ ok: true }`
* response headers include `Set-Cookie: semlox_session=...`
* Application → Cookies shows `semlox_session`
* user redirects to dashboard
* middleware accepts cookie
* local login still works
* production login works

Do not change anything else.
